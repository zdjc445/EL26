# 工业化开发生命周期

状态：书面评审中

版本：1.0

日期：2026-07-17

## 1. 目的

本规范将 Spec、Tests、Diff 和 Review 固化为强制质量门禁。目标不是增加形式，而是确保每项变更可追溯、可验证、可审查、可回滚，并且不依赖开发者或 Agent 的主观完成声明。

所有人类和 AI Agent 生成的变更执行同一流程。

## 2. 权威工件

每个变更由以下工件组成：

1. 产品需求或缺陷说明；
2. 已批准 Spec；
3. 必要的 ADR；
4. 实施计划；
5. 测试和评测证据；
6. 代码与迁移 Diff；
7. 独立 Review 记录；
8. 发布和回滚记录；
9. 生产指标或事故复盘。

文档状态只允许：

- `草案`：可以修改，不允许据此实现；
- `书面评审中`：等待明确批准，不允许据此实现；
- `已批准`：可以进入实施计划；
- `已替代`：保留历史并链接替代文档；
- `已废弃`：不再适用，但不能删除审计历史。

## 3. 角色与职责

同一成员可以承担多个角色，但单个高风险变更的实现者不能成为唯一 Reviewer。

| 角色 | 责任 |
|---|---|
| 需求提出者 | 说明问题、业务价值和成功条件 |
| Spec Owner | 消除歧义、维护需求编号和范围 |
| 实现者 | 按计划小步实现、提供验证证据和自审 |
| Reviewer | 独立检查需求、架构、安全、测试和 Diff |
| Release Owner | 检查发布门禁、执行发布和回滚决策 |
| Incident Owner | 事故期间协调止损、恢复、沟通和复盘 |

AI Agent 可以承担调查、实现、测试或 Review 工作，但不能自行批准产品范围、风险例外或生产发布。

## 4. 工作项类型

- `Feature`：新增用户能力；
- `Bug`：已定义行为未满足；
- `Security`：权限、凭证、数据泄漏或供应链风险；
- `Reliability`：可用性、恢复、幂等、并发或容量；
- `Refactor`：行为保持不变的结构改进；
- `Operations`：基础设施、监控、备份或发布；
- `Documentation`：不改变系统行为的文档变更；
- `Research`：有明确问题、时间边界和决策产物的技术验证。

不能用 `Refactor` 名义夹带功能变化。研究工作必须产出可接受或拒绝的结论，不能无限延长。

## 5. Definition of Ready

工作项满足以下条件才可进入实施计划：

- 用户问题和价值明确；
- 范围和非目标明确；
- 验收条件具有可观察结果；
- 涉及的业务规则和权限明确；
- 依赖和外部接口已识别；
- 数据模型及迁移影响已说明；
- 异常、重试、删除和回滚路径已说明；
- 测试层级和必要评测已说明；
- 安全和隐私影响已评估；
- Spec 状态为已批准；
- 重大技术选择已由 ADR 接受。

缺少任何一项时，不允许用“先写代码再说”绕过。

## 6. Spec 管理

### 6.1 需求编号

产品行为使用稳定需求编号，例如 `AUTH-01`、`CAL-08`、`RAG-04`。测试、实施计划和 PR 引用这些编号，不能复制一份容易漂移的需求文本代替链接。

### 6.2 变更控制

实现期间发现需求不完整时：

1. 停止受影响实现；
2. 更新 Spec；
3. 重新评审影响；
4. 调整测试和计划；
5. Spec 再次批准后继续。

禁止在代码中偷偷决定产品行为。

### 6.3 ADR

以下变化必须新增或替代 ADR：

- 核心框架或运行时；
- 数据库、检索引擎、消息系统；
- 服务边界；
- 外部供应商主链路；
- 身份和权限模型；
- 数据保留、加密或跨地域处理；
- 不能轻易逆转的工程决策。

ADR 记录背景、决策、备选方案、后果和替代条件。ADR 不记录普通编码偏好。

## 7. 实施计划门禁

实施计划在任何代码变更之前完成，并包含：

- 任务顺序和依赖；
- 将要创建或修改的准确文件；
- 每一步对应的需求编号；
- 先写哪些测试以及预期失败原因；
- 实现步骤；
- 每一步验证命令；
- 数据迁移和兼容顺序；
- 发布、监控和回滚步骤；
- 不在本次处理的内容。

计划按可独立验证的小任务拆分。一个任务不能同时横跨多个无关领域。

## 8. 测试驱动实现

功能和缺陷默认执行以下循环：

```text
根据 Spec 写测试
→ 运行并确认测试因缺少行为而失败
→ 写最小实现
→ 运行相关测试直至通过
→ 重构
→ 运行完整相关验证
```

实现者不得通过放宽断言、删除边界用例或修改测试数据掩盖错误。测试需求确实错误时，先更新 Spec 和测试理由，再修改实现。

数据库迁移、基础设施和可观测性变更无法完全使用普通 TDD 时，必须先定义可执行验证和回滚演练。

## 9. 分支与提交

- `main` 始终保持可发布；
- 变更使用短生命周期分支；
- 禁止绕过 Review 直接向 `main` 推送；
- 每个提交保持单一目的并能通过对应的基础检查；
- 提交说明使用 `<type>(<scope>): <summary>` 结构；
- 数据库迁移与依赖锁文件不得在 Review 中隐藏；
- 生成文件必须由可重复命令产生；
- 禁止提交密钥、验证码、真实用户资料和未脱敏生产日志。

合并策略由仓库保护规则固定，不能由单个 PR 临时改变。历史整理不能篡改已经发布的生产审计标签。

## 10. Diff 管理

每个 Diff 只解决一个可描述问题：

- 关联一个已批准 Spec 或明确缺陷；
- 不夹带无关重构；
- 不顺手升级无关依赖；
- 不改变未说明的公共接口；
- 新依赖说明必要性、许可证、维护状态和安全影响；
- 数据模型变化同时提供迁移、兼容和恢复证据；
- 大 Diff 按领域、迁移阶段或可独立发布能力拆分；
- 文件职责明显扩大时优先拆分，而不是继续堆叠。

不设机械行数上限。Reviewer 无法在一个连贯上下文中理解和验证时，Diff 必须拆分。

## 11. 自审

实现者在请求 Review 前逐项检查：

- 是否满足每条需求和非目标；
- 是否读取了实际文件、接口和 Schema，而非猜测标识符；
- 是否存在跨用户路径；
- 是否处理并发、幂等、重试和部分失败；
- 是否覆盖删除与恢复；
- 是否包含未使用代码或提前实现未来能力；
- 是否泄漏敏感数据；
- 是否更新相关文档；
- 是否运行最新完整验证；
- Diff 中是否出现意外文件、调试输出或供应商密钥。

自审结论和验证命令输出写入 PR，不用“应该没问题”等主观表述。

## 12. 独立 Review

Reviewer 使用未参与实现的独立上下文检查：

1. Spec 一致性；
2. 架构边界；
3. 权限和数据隔离；
4. 状态机、事务、并发和幂等；
5. 错误处理、补偿和恢复；
6. 测试是否能在实现错误时失败；
7. 日历时区和重复规则；
8. RAG 来源、注入和越权；
9. 依赖、许可证和供应链；
10. 可观测性、发布与回滚。

Review 反馈必须包含证据、影响和可定位位置。实现者验证反馈后修改，不能盲目接受，也不能无证据拒绝。

修复 Review 问题后重新运行受影响验证，并由 Reviewer 确认阻断项已关闭。

## 13. CI 门禁

合并前必须通过：

- 格式检查；
- 静态检查；
- 类型检查；
- 单元测试；
- 集成测试；
- 数据库迁移检查；
- API 契约检查；
- 依赖漏洞和许可证检查；
- 密钥扫描；
- 容器构建；
- RAG 固定评测；
- 关键浏览器端到端测试。

需要跳过检查时，只能通过有期限、记录责任人和补偿计划的风险例外；权限、安全、迁移和跨用户隔离检查不可跳过。

## 14. AI Agent 开发规则

- Agent 开始工作前读取当前 Spec、ADR、实际代码和测试；
- Agent 不得猜测路径、字段、环境变量、API 参数或数据库结构；
- Agent 不得自行扩大范围或执行未授权外部变更；
- Agent 编写实现前先编写或明确测试；
- Agent 必须报告实际运行的命令和结果；
- Agent 不得以自己的总结替代 Diff Review；
- Agent 生成代码与人工代码执行相同安全、许可证和质量检查；
- 高风险修改使用独立 Agent 或人工 Reviewer 复核；
- Agent 输出的模型建议不能成为生产配置，必须由评测和审批决定。

## 15. Definition of Done

只有以下全部成立才可报告完成：

- Spec 验收条件逐项满足；
- 最新完整验证通过；
- TDD 或等效验证证据存在；
- Diff 已自审；
- 独立 Review 已完成；
- CI 必需检查通过；
- 数据迁移和回滚路径验证；
- 文档同步更新；
- 预发布完成真实流程验收；
- 监控和告警已经生效；
- 没有未披露失败、跳过项或高风险问题。

“代码已写完”“测试大概能过”或“Agent 表示成功”都不满足完成定义。

## 16. 发布后管理

- 观察发布指标和错误预算；
- 异常达到回滚阈值时优先恢复服务；
- 事故记录时间线、用户影响、技术原因和处置；
- 复盘必须转化为 Spec、测试、告警或流程改进；
- 不将“操作失误”作为终止分析的根因；
- 重复性人工操作应进入自动化改进队列。

## 17. 风险例外

风险例外必须记录：

- 被跳过的规则；
- 原因和证据；
- 用户和系统影响；
- 补偿控制；
- 责任人；
- 到期时间；
- 关闭条件。

例外到期未关闭时阻止后续发布。零容忍项不得获得例外：跨用户访问、明文密钥、未经批准的高影响日历操作和不可恢复的未备份数据迁移。
