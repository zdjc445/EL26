# 测试与评测策略

状态：书面评审中

版本：1.0

日期：2026-07-17

## 1. 目标

测试体系同时验证确定性软件、非确定性 AI 行为、外部供应商契约、容量和故障恢复。任何单一层级通过都不能证明系统可发布。

## 2. 测试原则

- 测试从产品 Spec 和 ADR 派生；
- 测试验证公开行为，不绑定无关内部实现；
- 每个 Bug 修复包含能复现原问题的回归测试；
- 时间、随机数、外部服务和模型调用在单元测试中可控；
- 集成测试使用真实协议和临时依赖，不用无行为的空 Mock 替代；
- AI 评测使用固定、版本化、人工标注的数据集；
- 任何跨用户访问结果立即阻断发布；
- 不通过重复运行掩盖 Flaky 测试。

## 3. 需求追踪

每个测试用例或测试组标注对应需求编号。CI 生成需求到测试的追踪报告，至少覆盖：

- AUTH；
- CAL；
- LEARN；
- KB；
- CHAT；
- RAG；
- UI；
- SLO 和恢复目标。

新增需求没有测试映射时，Spec 不满足 Definition of Ready。

## 4. 测试层级

### 4.1 单元测试

目标：毫秒级验证纯业务规则。

范围：

- 认证状态转换；
- 验证码挑战规则；
- 日历时间区间；
- 重复规则和例外；
- 高影响操作分类；
- 学习任务状态；
- 文件和资料版本状态机；
- 内容分块；
- 整理提案状态；
- 来源优先级；
- 检索融合输入输出；
- 工具参数校验。

单元测试不访问网络、真实数据库或外部模型。

### 4.2 属性测试

使用生成式输入验证：

- 时区和夏令时边界；
- 重复事件的日期范围；
- 零长度、跨日和重叠时间；
- “本次／本次及后续／全部”系列变更；
- 幂等请求重复、乱序和并发；
- 分块长度和来源覆盖；
- 文件夹树循环和深度边界。

属性失败用例必须固化为回归样例。

### 4.3 数据库集成测试

使用临时 PostgreSQL 验证：

- 约束、索引和事务；
- 用户隔离；
- 并发版本检查；
- Outbox 原子性；
- 批量排期全有或全无；
- 迁移从上一生产版本升级；
- 回滚或向前修复路径；
- 删除工作流和引用关系。

SQLite 不能替代 PostgreSQL 集成测试。

### 4.4 基础设施集成测试

使用与生产协议一致的临时依赖验证：

- S3 兼容对象存储的分片上传、完成和删除；
- RabbitMQ 投递、确认、重投和失败队列；
- Celery Worker 崩溃后的恢复；
- Valkey 限流和缓存失效；
- Qdrant 稠密、稀疏、过滤、RRF 和索引版本切换；
- SSE 断线重连和重复事件处理。

### 4.5 外部供应商契约测试

针对阿里云百炼和验证码供应商验证：

- 请求和响应 Schema；
- 流式事件顺序；
- Tool Calling 参数；
- 超时和限流错误；
- 模型标识和用量字段；
- Embedding 维度；
- Rerank 返回数量和映射；
- 供应商错误码分类；
- 日志脱敏。

CI 默认使用受控替身和录制的脱敏契约。定时合成环境调用真实供应商，发现契约漂移立即告警。

### 4.6 端到端测试

浏览器端覆盖：

1. 邮箱注册和验证码登录；
2. 手机号注册和验证码登录；
3. 设置密码和密码登录；
4. 上传 PDF／DOCX；
5. 查看处理进度和失败原因；
6. 批准文件整理提案；
7. 指定资料范围并学习；
8. 查看行内引用；
9. 批准会话整理提案；
10. 创建学习任务和批量排期；
11. 查询、修改、删除和撤销日历操作；
12. 双侧栏折叠和窄屏抽屉；
13. 浏览器提醒；
14. 退出登录和会话撤销。

关键流程在预发布环境使用真实基础设施执行。

## 5. RAG 评测集

### 5.1 数据组成

评测集从经过授权和脱敏的大学资料构建，至少包含：

- 中文教材；
- 中英文混合材料；
- 具有标题层级的 DOCX；
- 多页 PDF；
- 表格文本；
- 相近术语；
- 精确关键词、章节和页码查询；
- 跨文档比较；
- 资料中不存在答案的问题；
- 包含提示词注入文本的资料；
- 两个用户拥有相似资料的隔离场景。

评测样本划分为开发集和发布门禁集。发布门禁集不得用于人工调整单个答案。

### 5.2 标注

每个问题记录：

- 允许检索的用户和范围；
- 相关资料版本；
- 相关内容块；
- 支持结论的最小证据；
- 不应出现的资料；
- 可以接受的通用知识补充；
- 应拒答或说明资料不足的条件。

### 5.3 发布指标

| 指标 | 首版门槛 |
|---|---:|
| 跨用户检索、引用或回答 | 0 次 |
| Recall@10 | ≥ 85% |
| 引用支持对应陈述 | ≥ 95% |
| 资料不足时不伪造资料依据 | ≥ 95% |
| 日历 Tool 参数 Schema 合法率 | 100% |
| 高影响操作未经批准执行 | 0 次 |

质量门槛同时按总体、资料类型和主要课程类别查看。总体通过但某个重要切片严重退化时不能发布。

### 5.4 评测版本

每次评测记录：

- 数据集版本；
- 解析器版本；
- 分块配置；
- Embedding 模型和维度；
- 稀疏检索配置；
- 融合配置；
- Rerank 模型；
- LLM 快照；
- 提示词和 Tool Schema 版本；
- 结果和运行时间。

任何一项变化都触发完整相关评测。

## 6. Agent 测试

### 6.1 状态图

测试所有节点、边和停止条件：

- 学习意图；
- 日历查询；
- 明确单事件创建；
- 歧义追问；
- 高影响操作中断；
- 批准、编辑和拒绝；
- 恢复执行；
- 模型拒绝 Tool；
- 达到步数或调用预算；
- 供应商失败。

### 6.2 幂等

LangGraph 中断后恢复会重跑节点。测试必须证明：

- 中断前没有非幂等写入；
- 重复恢复不会重复创建事件；
- Tool 超时后的重试不会重复提交事务；
- 相同调用标识返回首次结果；
- 审计记录保持完整且不重复计费。

### 6.3 模型替身

确定性测试使用可编程模型替身，精确输出 Tool 调用、无效参数、拒绝和超时。不能通过真实模型的偶然表现验证业务状态机。

## 7. 安全测试

- 修改 URL 或对象标识访问其他用户数据；
- 在文件夹移动、下载、索引和引用中绕过用户过滤；
- 预签名 URL 越权和过期；
- 验证码暴力尝试和发送轰炸；
- 会话固定和撤销后重用；
- CSRF、XSS、注入和恶意文件名；
- 文档提示词注入要求 Agent 调用日历工具；
- 日志、Trace、错误页和模型记录中的敏感数据；
- 依赖漏洞、许可证和密钥扫描。

跨用户测试必须覆盖每一个仓储和检索入口，不能只覆盖 HTTP 路由。

## 8. 性能与容量测试

### 8.1 发布容量基线

- 1000 个注册账号；
- 200 个同时在线会话；
- 50 个并发 Agent 流；
- 20 个同时文件上传；
- 50 个同时纯文本文档处理任务；
- 队列积压后恢复；
- 大用户与小用户之间的性能隔离。

### 8.2 延迟目标

| 操作 | p95 |
|---|---:|
| 普通 API | 500 ms |
| 日历查询与单事件写入 | 800 ms |
| 混合检索与重排 | 1.5 s |
| Agent 首个流式 Token | 5 s |
| 上传会话创建 | 1 s |
| 页面主要内容可交互 | 2.5 s |
| 提醒投递延迟 | 60 s |

负载测试分别报告服务内部时间和第三方供应商时间。

### 8.3 文档处理

- 上传确认后 60 秒内开始；
- 100 MB 以内可解析文档，p95 五分钟内可检索；
- 解析失败五分钟内显示明确状态；
- 重试不能产生重复版本或索引。

## 9. 故障与恢复测试

定期注入：

- API 实例退出；
- Worker 在任务中间退出；
- RabbitMQ 短暂不可用；
- PostgreSQL 连接耗尽；
- Qdrant 写入或查询失败；
- 对象存储超时；
- 模型限流或流式中断；
- 索引构建完成前失败；
- 提醒 Dispatcher 重启；
- 部署中途失败。

验证系统能恢复、任务不丢失、写入不重复、上一索引继续可用，并向用户显示真实状态。

## 10. 迁移测试

- 从上一生产 Schema 的备份副本执行迁移；
- 验证迁移耗时、锁范围和磁盘增长；
- 新旧应用在兼容阶段同时运行；
- 回滚应用时数据仍可读取；
- 破坏性 Contract 步骤只在旧版本完全退出后执行；
- 迁移失败有向前修复方案，不依赖未经验证的逆向迁移。

## 11. Flaky 测试管理

测试只要出现非预期不稳定就视为缺陷：

1. 记录失败证据；
2. 隔离根因；
3. 在限定时间内修复；
4. 临时隔离必须关联责任人和到期时间；
5. 隔离的测试不能被计入质量通过率。

禁止使用自动重跑把不稳定测试变成绿色。重跑只能用于收集诊断证据。

## 12. 测试命令接口

项目脚手架阶段必须定义一组跨本地和 CI 一致的仓库级命令，覆盖：

- 格式；
- 静态检查；
- 类型检查；
- 单元测试；
- 集成测试；
- 端到端测试；
- RAG 评测；
- 完整验证。

在脚手架创建前仓库不存在可执行命令，因此本设计不虚构命令名。实施计划必须先确定准确工具和命令，再创建代码；CI 和开发文档只引用实际存在且验证过的命令。
