# Time V1 工业化交付路线图

状态：书面评审中

版本：1.0

日期：2026-07-17

## 1. 目的

本路线图把已批准的产品规格、系统设计、ADR、测试策略和发布标准转换为可治理的交付序列。它只定义阶段边界、依赖、门禁和需求追踪，不替代每个阶段在开工前必须完成的详细实施计划。

产品展示名固定为 `Time`。后端 Python 包名固定为 `time_agent`，避免与 Python 标准库模块 `time` 冲突。代码托管仓库为 `git@github.com:zdjc445/EL26.git`，持续集成采用 GitHub Actions。

## 2. 管理方法

采用滚动式详细规划：

1. V1 全局范围由已批准 Spec 和 ADR 固定；
2. 阶段顺序和退出条件由本路线图固定；
3. 每个阶段只在其依赖已经落地、实际路径和接口可以读取后编写详细计划；
4. 每个详细计划列出准确文件、接口、失败测试、实现、验证和提交；
5. 每个任务执行 `Spec → Tests → Diff → Review`；
6. 当前阶段未通过独立 Review 和 CI，不启动依赖它的下一阶段；
7. 需求变化先更新 Spec，再更新路线图或实施计划，不能在代码中隐式决定。

不在本路线图中承诺工期。工期需要基于实际团队人数、Reviewer 可用性、供应商接入周期和阶段 0 得到的交付速率单独制定；在这些数据不存在时填写日期属于虚假精度。

## 3. 分支、工作项与 Review

- `main` 保持可发布；
- 每个阶段使用短生命周期分支，分支名采用 `phase/<phase-id>-<scope>`；
- 每个任务对应一个可独立 Review 的提交；
- PR 标题引用阶段编号，正文引用需求编号、计划文件、验证证据和回滚方式；
- 实现者不能成为高风险变更的唯一 Reviewer；
- 跨用户访问、未经批准的高影响日历写入、密钥泄漏和不可恢复迁移为零容忍阻断项；
- GitHub 分支保护、必需检查和 CODEOWNERS 在阶段 0 建立；
- GitHub 仓库设置属于外部状态，自动化配置前必须读取仓库的实际权限和规则，不能在文档中假定已启用。

## 5. 阶段详细计划规则

每个阶段的实施计划必须放在 `docs/superpowers/plans/`，并包含：

- 目标、架构、技术栈和全局约束；
- 准确创建／修改／测试文件路径；
- 对外接口、函数签名、字段名和事件名；
- 每个任务消费和产出的接口；
- 先写失败测试、确认失败原因、最小实现、通过验证、重构和提交；
- 数据迁移的 Expand／Migrate／Contract 顺序；
- 供应商替身、真实契约测试和失败分类；
- 安全、多租户、幂等、并发、删除和恢复验证；
- CI 检查、部署观察指标和回滚触发条件；
- 明确不在该阶段实现的能力。

阶段计划只能引用仓库中实际存在的标识符。尚未建立的外部供应商字段、资源名、GitHub 规则标识或 Vultr 资源标识，必须先通过研究工作项读取官方契约或实际控制面，再进入计划。

## 6. 阶段门禁

### 6.1 Ready

阶段进入实现前必须同时满足：

- 关联 Spec 和 ADR 为 `已批准`；
- 上游阶段退出门禁通过；
- 详细实施计划通过书面 Review；
- 需求到测试的追踪矩阵完整；
- 外部供应商和基础设施契约已经读取并固定；
- 数据模型、迁移、删除和恢复路径明确；
- 分支、实现者、独立 Reviewer 和 Release Owner 已在工作项中记录；
- 不存在未关闭的零容忍问题。

### 6.2 每个任务

每个任务按以下顺序执行：

```text
读取 Spec、ADR、计划和现有代码
→ 写失败测试或可执行验证
→ 确认失败原因正确
→ 写最小实现
→ 运行相关验证
→ 自审 Diff
→ 独立 Review
→ 修复并复验
→ 单一目的提交
```

### 6.3 阶段退出

- 需求验收逐项有证据；
- 格式、静态检查、类型检查、单元、集成、契约、E2E 和本阶段适用评测通过；
- 迁移和回滚已演练；
- 容器能够从锁文件重复构建；
- 依赖、镜像、许可证和密钥扫描通过；
- 独立 Review 阻断项关闭；
- 文档、OpenAPI 和运行手册同步；
- 工作区不存在未披露改动；
- 后续阶段需要的接口已经版本化。

## 8. 研究与 ADR 队列

以下工作必须产生有结论的 ADR 或契约记录，不能直接变成生产配置：

1. 邮箱验证码供应商、短信验证码供应商及错误码；
2. 阿里云百炼 LLM 的固定快照、Tool Calling 和流式事件；
3. `text-embedding-v4` 的实际维度、批量限制和费用字段；
4. `qwen3-rerank` 的请求限制、返回映射和失败分类；
5. Qdrant 中文稀疏表示能否达到 Recall@10 ≥ 85%；
6. Vultr 日本地域实际可用的计算、托管 PostgreSQL、网络和对象存储能力；
7. GitHub 仓库分支保护、环境审批和 Actions 权限。

研究任务必须有问题、输入数据、停止条件、结果和接受／拒绝结论。若结论改变已批准架构，先新增或替代 ADR。

## 9. V1 项目级完成定义

V1 只有在以下条件全部成立时才能宣布完成：

- 产品规格第 12 节的十步真实用户闭环全部通过；
- 全部需求在追踪矩阵中存在测试证据；
- 跨用户检索、下载、引用和回答为 0 次；
- Recall@10、引用支持率和低证据处理达到已批准门槛；
- 高影响日历操作未经批准执行为 0 次；
- 1000 用户容量基线和全部 p95 目标通过；
- PostgreSQL、对象存储和 Qdrant 恢复演练达到 RPO/RTO；
- 预发布真实基础设施的核心 E2E 通过；
- 独立 Review、发布审批、监控、告警和 Runbook 生效；
- 没有未披露跳过项、过期风险例外或未关闭零容忍问题。
