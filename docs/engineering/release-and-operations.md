# 发布与运维标准

状态：书面评审中

版本：1.0

日期：2026-07-17

## 1. 目标

生产发布必须可重复、可观察、可停止和可恢复。任何生产变更都通过版本化代码、基础设施即代码和 CI/CD 执行，禁止依赖未记录的控制台手工操作作为正常发布流程。

## 2. 环境

### 本地

用于开发和快速测试。只使用合成或脱敏数据，不能连接生产数据库、生产对象存储或生产供应商密钥。

### CI

每次变更创建隔离依赖，运行静态检查、测试、迁移、契约、镜像构建和安全扫描。CI 凭证遵循最小权限并定期轮换。

### 预发布

拓扑和配置结构与生产一致，规模可以不同。使用独立数据库、Bucket、Qdrant 集合、消息系统和供应商工作空间。执行真实端到端、迁移、负载抽样和发布演练。

### 生产

只接受经过审批的不可变镜像和 Terraform 变更。生产数据不能复制回低环境，除非完成授权和不可逆脱敏。

## 3. 基础设施管理

- Vultr 资源由 Terraform 管理；
- Terraform State 使用受保护远端存储和锁；
- 资源名称、地域、网络、访问策略和标签来自版本化配置；
- 手工紧急变更在恢复后立即回写代码并审计漂移；
- 定期执行 Plan 检测漂移；
- 生产网络默认拒绝，数据库、RabbitMQ、Valkey 和 Qdrant 不暴露公网；
- 管理访问使用独立身份、最小权限和审计；
- 对象存储 Bucket 默认私有。

## 4. 构建与供应链

- 前后端依赖使用锁文件；
- 基础镜像固定不可变摘要；
- 构建生成 SBOM；
- 扫描依赖、镜像和许可证；
- 镜像签名后进入制品库；
- 同一制品从预发布晋升到生产，不在生产重新构建；
- 供应商 SDK 升级视为契约变化，运行对应集成测试；
- 高危漏洞没有有效缓解措施时阻止发布。

## 5. 发布流水线

```text
合并到 main
→ 完整 CI
→ 构建并签名制品
→ 部署预发布
→ 数据库迁移演练
→ 冒烟与关键 E2E
→ RAG 固定评测
→ 发布审批
→ 生产兼容迁移
→ 滚动部署
→ 健康与业务指标验证
→ 完成或回滚
```

每次发布记录：

- Git 提交；
- 镜像摘要；
- Terraform Plan；
- 迁移版本；
- Spec 和 ADR；
- 测试与评测结果；
- 审批人；
- 开始和完成时间；
- 监控链接；
- 回滚点。

## 6. 数据库迁移

使用 Expand／Migrate／Contract：

1. Expand：添加向前兼容 Schema；
2. 部署可读写新旧结构的应用；
3. Migrate：受控迁移历史数据并验证；
4. 切换所有读写；
5. 观察至少一个完整发布周期；
6. Contract：确认旧版本退出后删除旧结构。

禁止在同一次发布中先删除旧字段再部署新代码。大表变化先测量锁和磁盘影响。生产失败优先向前修复；逆向迁移必须预先在真实规模副本上验证才能作为回滚方案。

## 7. 应用发布

- 在线 API 至少两个实例；
- 使用就绪和存活检查；
- 实例停止接收新请求后完成在途请求再退出；
- SSE 断开后客户端可安全重连；
- Worker 按队列滚动停止，未确认任务由 Broker 重投；
- 调度进程使用租约保证单一有效执行者；
- 新索引完整验证后原子切换；
- LangGraph Checkpoint 与应用版本兼容，不兼容变化必须提供迁移。

## 8. 回滚

触发条件包括：

- 错误率或延迟超过发布阈值；
- 跨用户权限异常；
- 数据写入错误；
- 消息重复或丢失；
- RAG 质量低于发布门槛；
- Agent 未经批准执行高影响操作；
- 关键依赖无法稳定调用。

回滚步骤：

1. 停止继续放量；
2. 禁止有风险写操作；
3. 恢复上一应用镜像或关闭功能开关；
4. 保持兼容 Schema；
5. 验证核心流程和队列；
6. 评估错误数据并执行经批准的补偿；
7. 记录事故并开始复盘。

功能开关不能长期代替删除废弃代码。每个开关具有 Owner、默认值、到期条件和清理任务。

## 9. SLO 与错误预算

### 9.1 月度可用性

| 能力 | SLO |
|---|---:|
| 登录、日历、知识空间和文件管理 API | 99.9% |
| Agent 对话入口 | 99.5% |
| 异步资料处理 | 99.5% 最终成功或明确永久失败 |

99.9% 月度可用性对应约 43.8 分钟不可用时间。计划维护只有在用户不受影响时才排除；实际排除规则必须在监控配置中固定。

### 9.2 延迟

| 操作 | p95 |
|---|---:|
| 普通 API | 500 ms |
| 日历查询和单事件写入 | 800 ms |
| 混合检索与重排 | 1.5 s |
| Agent 首个流式 Token | 5 s |
| 上传会话创建 | 1 s |
| 页面主要内容可交互 | 2.5 s |
| 提醒投递延迟 | 60 s |

第三方供应商时间单独记录，同时计入用户实际体验 SLI。

### 9.3 错误预算策略

- 预算消耗正常：允许功能发布；
- 快速消耗：暂停非可靠性变更，优先修复；
- 预算耗尽：停止常规发布，只允许安全、恢复和可靠性变更；
- 跨用户泄漏和未经批准的高影响操作不使用错误预算，发生一次即进入事故流程。

## 10. 容量基线

发布前验证：

- 1000 个注册账号；
- 200 个同时在线会话；
- 50 个并发 Agent 流；
- 20 个同时上传；
- 50 个并发纯文档处理任务；
- 队列积压恢复；
- 大用户不会造成其他用户明显性能退化。

实例规格和扩容阈值由负载测试数据决定。生产按 CPU、内存、连接池、队列延迟、检索延迟和供应商限流共同扩容，不能只看请求数。

## 11. 备份和恢复

### 11.1 目标

| 数据 | RPO | RTO |
|---|---:|---:|
| PostgreSQL 业务数据 | 15 分钟 | 2 小时 |
| 原始资料和派生对象 | 15 分钟 | 4 小时 |
| Qdrant 检索索引 | 24 小时 | 8 小时 |
| 缓存和临时协调状态 | 允许丢失 | 30 分钟 |

### 11.2 PostgreSQL

- 选择满足 RPO/RTO 和时间点恢复窗口的 Vultr Managed PostgreSQL 计划；
- 自动备份外，定期生成独立逻辑备份；
- 备份加密并限制访问；
- 每季度恢复到隔离环境并运行数据一致性检查；
- 未验证恢复的备份不能视为有效。

### 11.3 对象存储

- 启用对象版本和生命周期规则；
- 已确认上传的对象异步复制到独立备份位置，持续监控复制延迟；
- 备份凭证与生产写凭证分离；
- 删除工作流使用延迟清理窗口，防止误删除立即扩散；
- 每季度抽样恢复原始文件和解析结果。

Vultr Object Storage 当前不支持 Bucket Replication，因此复制由受控任务实现，不能假设平台自动完成。

### 11.4 Qdrant

- 周期性快照；
- 快照存入独立备份位置；
- 保留从 PostgreSQL 内容块和固定模型版本完整重建索引的能力；
- 恢复时先验证用户过滤、点数量和抽样查询，再切换流量。

## 12. 监控

### 平台指标

- CPU、内存、磁盘、网络；
- 数据库连接、慢查询、锁、复制和容量；
- RabbitMQ 队列深度、最老消息、重投和失败队列；
- Qdrant 查询、写入、段合并、内存和磁盘；
- Object Storage 容量、请求和复制延迟。

### 应用指标

- 请求率、错误率和延迟；
- 在线 SSE 数；
- 登录和验证码异常；
- 文档处理阶段耗时和失败率；
- Agent 首 Token 和总耗时；
- 模型 Token、调用量、限流和成本；
- 检索空结果、Recall 抽样和低证据回答；
- 提醒延迟；
- 拒绝的越权访问；
- 删除清理积压。

### 告警

告警必须具有 Owner、严重级别、可操作描述和 Runbook。不能创建长期无人处理的噪声告警。

立即告警：

- 跨用户访问或引用；
- 数据库不可连接；
- 备份失败；
- 队列持续积压；
- 提醒延迟超过目标；
- 模型供应商连续失败；
- 对象复制超过 RPO；
- 存储接近容量限制；
- RAG 发布评测回归。

## 13. 成本治理

即使预算不设上限，也必须按用户和能力记录：

- 原始和派生存储；
- 上传和下载流量；
- 解析页数和处理时间；
- Embedding Token；
- Rerank Token；
- LLM 输入输出 Token；
- 检索和数据库资源；
- 通知发送量。

设置异常消费告警和每任务调用预算。无限重试、重复索引和异常循环必须被成本守卫阻断。

## 14. 事故管理

### 严重级别

- `SEV-1`：跨用户泄漏、凭证泄漏、不可恢复数据损坏、未经批准高影响操作或核心服务全面不可用；
- `SEV-2`：大量用户无法使用关键能力、持续数据处理失败或 SLO 快速消耗；
- `SEV-3`：局部功能退化、有替代路径且数据安全未受影响；
- `SEV-4`：低影响缺陷或运维改进。

### 响应流程

```text
发现
→ 定级
→ 指定 Incident Owner
→ 止损
→ 恢复
→ 验证
→ 用户沟通
→ 无责复盘
→ 跟踪改进项关闭
```

SEV-1 和 SEV-2 必须形成复盘，包含时间线、用户影响、检测缺口、根因、促成因素和可验证改进项。不能以“人为失误”结束根因分析。

## 15. 运行手册

生产上线前必须存在并演练：

- 数据库恢复；
- 对象恢复；
- Qdrant 重建和切换；
- RabbitMQ 积压和失败队列处理；
- 模型供应商故障；
- 验证码供应商故障；
- 误发布回滚；
- 数据删除失败；
- 密钥轮换；
- 跨用户安全事故。

Runbook 中只引用实际命令和资源标识。基础设施创建前不猜测名称；实施阶段从 Terraform 和控制台读取后填入。
