# ADR-0001：模块化单体与运行时拆分

状态：书面评审中

日期：2026-07-17

## 背景

首版面向约 1000 名大学生，业务包含认证、日历、学习、对话、知识空间、资料处理、RAG 和通知。原始资料为 TB 级，但重负载主要来自异步文档处理和模型调用，而不是所有领域都需要独立扩缩容。

## 决策

采用模块化单体代码库，运行时拆分为：

- Web 前端；
- 多实例 Python 在线 API；
- LangGraph 在线 Agent；
- Celery Worker 池；
- 提醒和任务调度进程；
- Qdrant 检索服务；
- RabbitMQ 和 Valkey；
- 托管 PostgreSQL 和 Object Storage。

领域模块通过公开应用接口交互，共享 PostgreSQL 事务，但禁止跨模块直接访问内部 ORM 模型。

## 备选方案

### 首版微服务和 VKE

优点：独立部署和扩缩容。

拒绝原因：首版会引入服务发现、契约版本、跨服务事务、分布式追踪和 Kubernetes 运维，不能直接提升学习体验或 RAG 质量。

### 单进程单实例

优点：最简单。

拒绝原因：在线请求、Agent 流和文档任务会互相干扰，无法满足可用性和独立扩容要求。

### AI 能力完全托管

优点：快速上线。

拒绝原因：用户数据、索引、删除和审计会绑定单一平台，难以保持业务控制权。

## 后果

正面：

- 保留简单事务边界；
- 在线与异步负载独立扩容；
- 本地和集成测试可控；
- Diff 和 Review 能按领域拆分；
- 以后可以逐个提取服务。

负面：

- 必须通过架构测试和 Review 防止模块耦合；
- 计算、RabbitMQ 和 Qdrant 需要运维；
- 共享数据库变化需要严格迁移纪律。

## 重新评估触发条件

满足任一条件时新增 ADR 评估服务拆分或 VKE：

- 一个领域需要完全独立的发布节奏和团队所有权；
- 单体发布频繁造成不可接受的变更冲突；
- 某模块无法通过独立进程和 Worker 扩容满足容量；
- 安全或合规要求物理隔离；
- 负载测试证明共享数据库或部署单元是明确瓶颈。
