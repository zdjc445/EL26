# ADR-0003：纯文本 RAG 与检索架构

状态：已批准

日期：2026-07-17

## 背景

用户上传数 GB 级学习资料，需要中文语义检索、精确关键词检索、资料范围过滤、引用和多租户隔离。首版已经明确不实现 OCR 或图片上传，只解析具有文本层的 PDF 和 DOCX。

## 决策

- PostgreSQL 保存权威内容块和业务元数据；
- Qdrant 保存可重建检索索引；
- 使用稠密向量与本地稀疏词项表示；
- 使用 RRF 融合召回结果；
- 使用 `qwen3-rerank` 重排；
- `text-embedding-v4` 是首版 Embedding 候选；
- 生产模型和维度经固定中文评测集选择并作为索引版本固定；
- 一套 Embedding 模型使用一套集合，以用户 Payload 分区，不按学生建立独立集合；
- 所有查询由后端强制用户、文件夹和资料范围过滤；
- 只有原始资料和用户批准的派生学习条目进入默认检索；
- 回答保留内容块级来源追踪和行内引用。

## 备选方案

### PostgreSQL 全文检索加 pgvector

优点：系统少、事务简单。

未选原因：Vultr 托管 PostgreSQL 可用扩展和中文分词能力必须从实际产品读取；首版资料规模需要独立检索扩缩容。

### OpenSearch

优点：成熟全文和向量混合检索。

未选原因：首版运维和资源复杂度更高。若 Qdrant 中文稀疏检索未达到发布门槛，OpenSearch 是上线前替代方案。

### 阿里云百炼托管知识库

优点：集成快。

拒绝原因：不能作为唯一业务和索引控制面，资料版本、删除、审计和供应商迁移必须由本系统控制。

## 质量门禁

- 跨用户结果：0；
- Recall@10：至少 85%；
- 引用支持对应陈述：至少 95%；
- 资料不足时不伪造资料依据：至少 95%；
- 中文关键词、专有名词、公式文本和章节号分别评测；
- Embedding、稀疏模型、维度或融合算法改变时重建新索引并重新评测。

## 后果

正面：

- 混合检索兼顾语义和精确词项；
- 索引可以独立扩容和重建；
- 用户过滤和来源追踪明确；
- 不绑定托管知识库。

负面：

- 需要运维 Qdrant；
- 中文稀疏表示必须通过真实资料验证；
- 索引发布、版本切换和删除需要应用层协调。

## OCR 扩展

首版解析器返回统一内容块接口。未来 OCR 通过新的解析器和资料派生版本接入，不修改现有资料身份，不让 OCR 结果覆盖原始文件或历史解析结果。OCR 不得在没有新 Spec 和评测集的情况下直接启用。
